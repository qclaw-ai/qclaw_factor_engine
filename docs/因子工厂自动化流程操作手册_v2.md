# 因子工厂自动化流程操作手册

# 前言

本手册用于规范「因子工厂自动化流程」的全环节操作，明确各步骤的核心目标、操作规范、输入输出标准及异常处理，确保流程可复现、可管控、不依赖人工干预，同时限定因子检索来源为指定3个专业量化平台，保障因子质量。

核心目标：实现「因子检索→回测验证→入库管理→定期复检→淘汰更新」的全自动化闭环，降低人工成本，保持因子库高质量、可追溯。

# 一、核心基础信息

## 1.1 因子检索指定来源（唯一合法来源）

- 专业交易因子知识库：https://factors.directory/zh（优先，结构化因子，易自动化提取）

- BigQuant论坛：https://bigquant.com/（补充，含大量中文因子思路）

- 聚宽：https://www.joinquant.com/（补充，含因子代码及回测案例）

说明：禁止从上述3个网站以外的渠道检索因子，避免无效、错误因子进入流程。

## 1.2 核心工具与环境

- 数据来源：AkShare（免费开源，获取全市场A股历史行情数据）

- 因子评估：Alphalens 或自研轻量 IC 计算模块（用于单因子分层收益、IC/IC_IR 等评估指标）

- 策略回测：Backtrader（用于后续多因子组合或策略层面的回测，不参与本版本单因子筛选 MVP 核心流程）

- 数据库：PostgreSQL（存储因子库、行情数据、回测结果及操作日志）

- 调度工具：APScheduler（定时触发因子复检流程，MVP 阶段可先通过脚本手动触发）

- 辅助工具：OpenClaw（仅用于因子检索、文档标准化，不参与核心计算）

- 脚本语言：Python（全流程自动化脚本开发）

## 1.3 核心定义与阈值（通过配置统一管理）

### 1.3.1 因子有效性判定阈值（用于步骤3筛选入库）

- IC值（因子与股价涨跌相关性）：≥0.08

- IC_IR（IC值稳定性）：≥1.5

- 夏普比率（收益风险比）：≥1.5

- 最大回撤（最大亏损比例）：≤0.15

- 换手率（调仓频率合理性）：≤0.3

说明：以上数值阈值不写死在代码中，统一存放在阈值配置（如 PostgreSQL 中的阈值配置表或独立 ini 配置文件）中，由流程在运行时读取当前生效版本的阈值进行判断。

### 1.3.2 因子过时判定阈值（用于步骤6判定）

- IC值衰减率：＞50%（衰减率=（初始IC值-最新IC值）/初始IC值）

- 最新IC值：＜0.08

- 连续复检不合格次数：≥2次

上述过时判定阈值同样来源于统一的阈值配置（数据库表或配置文件），支持版本化管理和追溯。

### 1.3.3 标准化格式定义

- 因子文档：Markdown格式（固定模板，见步骤1）

- 回测结果：JSON格式（固定字段，见步骤2）

- 日志记录：TXT格式（带时间戳，全流程留痕）

# 二、全流程详细操作步骤

## 步骤1：从指定网站检索因子并生成标准化文档

### 1.1 核心目标

从3个指定量化网站，检索公开、可靠的量化因子，提取核心信息并整理为标准化文档，为后续回测提供统一输入，同时过滤重复因子。

### 1.2 操作步骤

1. 配置检索规则：设定检索关键词（如“量价因子”“基本面因子”“动量因子”），限定检索范围为3个指定网站，排除广告、无效帖子。

2. 因子提取：通过OpenClaw或自动化脚本，从3个网站提取因子核心信息，需包含以下必填字段（缺一不可）：
        

    - 因子名称

    - 因子公式（使用统一的最小 DSL 表达式，可直接用于计算；仅支持 + / - / * / ÷ 四则运算，以及 MA(x, n)、REF(x, n) 等少量基础函数）

    - 因子描述（说明因子逻辑、预测方向）

    - 因子类型（动量/价值/成长/质量/波动率/量价等）

    - 适用股票池（全A/沪深300/中证500等）

    - 调仓周期（日/周/月）

    - 因子方向（factor_direction，long/short，用于标记因子是正向还是反向）

    - 来源URL（对应网站的因子原文链接，便于追溯）

3. 重复因子过滤：将提取的因子与已有因子库比对（通过因子名称、公式双重比对），避免重复录入。

4. 生成标准化文档：按固定Markdown模板，整理上述信息（包含因子方向等必填字段），生成因子文档（命名规则：因子ID_因子名称.md，因子ID格式：FACTOR_来源缩写_日期_序号，如FACTOR_FD_20260310_001）。

### 1.3 输入输出

- 输入：3个指定网站的因子相关内容（结构化公式、论坛帖子、代码片段）

- 输出：标准化因子Markdown文档（单个因子对应一个文档）、重复因子过滤日志

### 1.4 示例（3个网站分别举例）

### 1.5 异常处理

- 因子信息缺失：若提取的因子缺少必填字段（如无公式），自动标记为无效，不生成文档，记录日志并跳过。

- 网站无法访问：若某一网站无法访问，自动切换至其他两个网站，待网站恢复后补全检索，记录异常日志。

## 步骤2：根据因子文档运行回溯代码，测试因子并记录结果

### 2.1 核心目标

利用历史行情数据，根据步骤1生成的因子文档，计算因子值并执行回测，验证因子有效性，生成标准化回测结果，为因子入库提供依据。

### 2.2 操作步骤

1. 读取因子文档：自动化脚本批量读取步骤1生成的所有标准化因子文档，解析出因子公式、适用股票池、调仓周期等核心参数。

2. 加载历史行情数据：从PostgreSQL数据库的stock_daily表中，读取对应股票池（如沪深300）、对应周期（如日线）的历史数据（时间范围：2021年至今），数据字段包含：stock_code（股票代码）、trade_date（交易日期）、open（开盘价）、high（最高价）、low（最低价）、close（收盘价）、volume（成交量）、turnover（换手率）。

3. 计算因子值：根据因子文档中的公式，利用Python（Pandas/NumPy）为每只股票、每个交易日计算因子值，处理数据异常（如空值、异常值），确保计算结果准确。MVP 版本中，统一采用全市场截面 1%–99% 分位去极值 + 全市场 Z-score 标准化，不做行业/规模中性化。

4.执行因子回测：调用 Alphalens 工具或自研轻量 IC 计算模块，执行轻量回测，核心流程包括：
(1)每日按因子值对股票进行横截面排序
(2)分为 10 层分组，统计每层未来 N 日平均收益率（MVP 默认使用 5 日，即 horizon=5d，未来收益定义为 log(close_t+5 / close_t)）
(3)计算 IC 值（因子与未来收益的秩相关系数）、IC_IR（IC 值均值 / 标准差）
(4)统计分层收益差、换手率等指标

5. 记录回测结果：将回测指标整理为标准化JSON格式，与对应因子文档关联，生成回测结果文件（命名规则：因子ID_backtest.json），同时写入数据库回测表。

### 2.3 输入输出

- 输入：步骤1生成的标准化因子Markdown文档、PostgreSQL中的历史行情数据、回测基础模板脚本

- 输出：标准化回测结果JSON文件、回测日志（含计算过程、回测参数）

### 2.4 示例（回测结果JSON）

```Plain Text

{
  "factor_id": "FACTOR_FD_20260310_001",
  "factor_name": "20日动量",
  "backtest_period": "2021-01-01 至 2025-12-31",
  "test_universe": "全市场A股",
  "trading_cycle": "日线",
  "horizon": "5d",
  "key_metrics": {
    "ic_value": 0.12,
    "ic_ir": 2.1,
    "annual_return": 0.18,
    "max_drawdown": 0.09,
    "sharpe_ratio": 1.75,
    "turnover": 0.26
  },
  "pass_standard": true,
  "backtest_time": "2026-03-10 10:30:00",
  "log_path": "./logs/backtest/FACTOR_FD_20260310_001.log"
}
    
```

### 2.5 异常处理

- 数据缺失：若某只股票的历史数据缺失（如无收盘价），自动跳过该股票，不影响整体回测，记录缺失数据日志。

- 公式解析错误：若因子文档中的公式无法解析（如语法错误），标记该因子回测失败，记录错误日志，跳过该因子，继续执行其他因子回测。

- 回测报错：若回测过程中出现程序报错，自动终止该因子回测，记录报错信息，继续执行下一个因子，避免流程中断。

## 步骤3：根据记录结果选择因子入库

### 3.1 核心目标

根据步骤2的回测结果，对照预设的有效性阈值，筛选合格因子，正式存入因子库，建立因子全生命周期管理，确保因子库高质量。

### 3.2 操作步骤

1. 读取回测结果：自动化脚本批量读取步骤2生成的所有回测结果JSON文件，提取核心指标（IC值、夏普比率等）。

2. 筛选合格因子：将回测指标与1.3.1节的有效性阈值比对，全部满足阈值条件的因子判定为“合格”，否则为“不合格”。

3. 因子入库：将合格因子的相关信息写入PostgreSQL数据库的3张关联表（因子基础信息表、因子回测表、因子文件关联表），完成入库。
        

    - 因子基础信息表（factor_basic）：存储因子ID、名称、类型、适用股票池、调仓周期、来源URL、入库时间、是否有效等。

    - 因子回测表（factor_backtest）：存储因子ID、回测周期、核心指标、回测时间等，与基础信息表关联。

    - 因子文件关联表（factor_files）：存储因子ID、因子文档路径、回测结果文件路径、回测日志路径等。

4. 生成入库记录：记录每只合格因子的入库时间、操作人（系统自动标记）、入库状态，生成入库日志，确保可追溯。

5. 不合格因子处理：将不合格因子的文档、回测结果归档至指定目录，标记为“未入库”，记录原因（如IC值不足），不进入后续流程。

### 3.3 输入输出

- 输入：步骤2生成的回测结果JSON文件、因子有效性判定阈值配置文件

- 输出：因子库入库记录、入库日志、不合格因子归档文件

### 3.4 示例（数据库表记录）

#### 因子基础信息表（factor_basic）

|factor_id|factor_name|factor_type|test_universe|trading_cycle|source_url|create_time|is_valid|
|---|---|---|---|---|---|---|---|
|FACTOR_FD_20260310_001|20日动量|动量因子|全市场A股|日线|https://factors.directory/zh/momentum/20_day_momentum|2026-03-10 11:00:00|true|
### 3.5 异常处理

- 入库失败：若因数据库连接异常、字段缺失导致入库失败，自动重试3次，仍失败则记录错误日志，通知相关人员排查，不影响其他因子入库。

- 因子重复入库：入库前再次比对因子ID，避免重复入库，若发现重复，标记为异常，记录日志并跳过。

## 步骤4：定时触发

### 4.1 核心目标

按预设的时间规则，自动触发对因子库中所有有效因子的重新回测流程，实现因子有效性的定期监控，及时发现因子过时问题。

### 4.2 操作步骤

1. 配置定时规则：使用APScheduler配置定时任务，预设触发时间（推荐：每月1日凌晨00:00，避免影响日常操作），定时规则格式：0 0 1 * *（每月1日0点执行）。

2. 触发任务生成：定时任务触发时，自动化脚本从PostgreSQL因子库中读取所有is_valid = true（有效）的因子ID，生成批量回测任务清单（包含因子ID、回测时间范围等）。

3. 启动回测流程：脚本自动调用步骤2的回测脚本，将批量回测任务清单传入，启动批量回测，流程与步骤2一致，仅回测时间范围更新为“2021年至今”（包含最新行情数据）。

4. 触发日志记录：记录定时任务的触发时间、任务清单、执行状态，生成定时触发日志，便于后续追溯。

### 4.3 输入输出

- 输入：定时规则配置文件、因子库中所有有效因子信息、最新历史行情数据

- 输出：批量回测任务清单、定时触发日志、批量回测启动指令

### 4.4 示例（定时任务配置）

```Plain Text

{
  "trigger_type": "cron",
  "cron_rule": "0 0 1 * *",
  "task_name": "每月因子批量复检",
  "description": "每月1日凌晨00:00，对因子库中所有有效因子进行重新回测，验证有效性",
  "backtest_period": "2021-01-01 至 最新交易日",
  "target_factors": "all"（所有有效因子）
}
    
```

### 4.5 异常处理

- 定时任务未触发：若因系统故障、调度工具异常导致任务未触发，记录异常日志，自动补发任务，同时通知相关人员排查。

- 批量回测中断：若批量回测过程中出现中断，记录中断位置，重启后从中断位置继续执行，避免重复回测。

## 步骤5：对库内因子进行回溯

### 5.1 核心目标

复用步骤2的回测框架，使用包含最新行情数据的历史数据，对因子库中的有效因子进行重新回测，生成最新的有效性指标，为因子过时判定提供依据。

### 5.2 操作步骤

1. 加载最新行情数据：从PostgreSQL数据库的stock_daily表中，读取包含最新交易日的行情数据（确保数据已通过每日自动更新脚本更新）。

2. 批量回测执行：按照步骤4生成的批量回测任务清单，对每个有效因子，重复步骤2的“因子值计算→回测→指标统计”流程，回测时间范围包含最新数据。

3. 生成最新回测报告：为每个因子生成包含最新核心指标的回测报告（JSON格式，命名规则：因子ID_backtest_latest.json），与初始回测报告关联，写入数据库回测表（新增最新回测记录，保留历史记录）。

4. 回溯日志记录：记录每个因子的回溯时间、最新回测指标、执行状态，生成因子回溯日志，确保可追溯。

### 5.3 输入输出

- 输入：批量回测任务清单、最新历史行情数据、因子库中有效因子信息

- 输出：因子最新回测报告（JSON）、因子回溯日志、数据库回测记录更新

### 5.4 示例（最新回测报告JSON）

  "backtest_period": "2021-01-01 至 2026-03-31",
  "test_universe": "全市场A股",
  "trading_cycle": "日线",
  "key_metrics": {
    "ic_value": 0.05,
    "ic_ir": 0.8,
    "annual_return": 0.09,

### 5.5 异常处理

- 最新数据缺失：若最新交易日的行情数据未更新，自动延迟回溯任务，等待数据更新完成后再执行，记录延迟日志。

- 单个因子回溯失败：若某一因子回溯失败（如公式解析错误、数据缺失），标记该因子回溯失败，记录错误日志，继续执行其他因子回溯，不影响整体流程。

## 步骤6：因子是否过时？→ 剔除因子出库 / 保留

### 6.1 核心目标

对比因子的初始回测指标与步骤5生成的最新回测指标，根据预设的过时判定阈值，自动判定因子是否过时，对过时因子进行逻辑删除（出库），对有效因子更新最新回测指标，保持因子库活力。

### 6.2 操作步骤

1. 读取对比数据：从数据库中读取每个有效因子的初始回测指标（步骤2）和最新回测指标（步骤5），提取核心对比指标（IC值）。

2. 计算IC衰减率：按照公式计算因子IC值衰减率（衰减率=（初始IC值-最新IC值）/初始IC值）。

3. 判定因子状态：对照1.3.2节的过时判定阈值，判定因子为“过时”或“有效”：
       

    - 过时：满足任一条件（IC衰减率＞50%、最新IC值＜0.08、连续2次复检不合格）。

    - 有效：不满足任何过时条件，因子有效性未明显衰减。

4. 因子状态更新：

    - 过时因子：在数据库factor_basic表中，将is_valid字段更新为false（逻辑删除，不物理删除），在factor_lifecycle表中记录淘汰原因、淘汰时间、操作人。

    - 有效因子：在数据库factor_backtest表中，用最新回测指标更新该因子的回测记录，在factor_lifecycle表中记录更新时间、操作人。

5. 生成状态更新报告：整理所有因子的判定结果、状态更新情况，生成因子状态更新报告，便于监控因子库状态。

### 6.3 输入输出

- 输入：因子初始回测指标、最新回测指标、因子过时判定阈值配置文件

- 输出：因子状态更新报告、因子生命周期记录、数据库因子状态更新

### 6.4 示例（因子过时判定过程）

- 因子ID：FACTOR_FD_20260310_001（20日动量）

- 初始IC值：0.12，最新IC值：0.05

- IC衰减率：(0.12-0.05)/0.12≈58.3%＞50%

### 6.5 异常处理

- 判定逻辑错误：若因指标读取错误、衰减率计算错误导致判定异常，记录错误日志，手动复核该因子，重新判定并更新状态。

- 状态更新失败：若因数据库异常导致状态更新失败，自动重试3次，仍失败则记录错误日志，通知相关人员排查。

# 三、数据管理规范

## 3.1 行情数据管理

- 数据来源：仅使用AkShare获取全市场A股日线数据，禁止使用其他来源数据。

- 更新频率：每日凌晨2:00，自动更新前一交易日的行情数据，写入PostgreSQL的stock_daily表。

- 数据备份：每周日凌晨，对stock_daily表进行全量备份，防止数据丢失。

- 数据清理：每月月底，清理异常数据（如收盘价为0、成交量异常过大的数据），记录清理日志。

## 3.2 因子文档与回测结果管理

- 存储路径：统一存储在指定目录（如./factor_docs/、./backtest_results/），按因子ID分类归档。

- 备份策略：每月备份一次所有因子文档、回测结果文件，与数据库备份同步。

- 归档规则：未入库因子、过时因子的相关文件，归档至指定归档目录，保留1年，便于追溯。

## 3.3 日志管理

- 日志类型：包含因子检索日志、回测日志、入库日志、定时触发日志、回溯日志、状态更新日志、异常日志。

- 存储路径：统一存储在./logs/目录，按日志类型分类，命名格式：日志类型_日期.log。

- 保留期限：日志保留6个月，超过期限自动清理，核心日志（入库、淘汰记录）保留1年。

# 四、异常处理总则

1. 全流程异常均需记录日志，包含异常时间、异常位置、异常原因、处理方式，确保可追溯。

2. 单个因子的异常（如回测失败、入库失败），不影响整体流程，自动跳过该因子，继续执行其他因子。

3. 流程级异常（如数据库崩溃、定时任务失效），自动终止流程，记录异常日志，通知相关人员排查，排查完成后手动重启流程。

4. 异常处理完成后，需补充相关记录，更新因子/数据状态，确保流程连续性。
