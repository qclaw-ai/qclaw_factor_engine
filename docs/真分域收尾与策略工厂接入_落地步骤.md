# 真分域收尾与策略工厂接入 — 落地步骤

> 目的：在 **因子工厂（qclaw_factor_engine）** 真分域已大体落地的前提下，把 **剩余缺口** 和 **策略工厂（qclaw_strategy_engine）** 的读盘契约钉死，并按优先级逐项实施。  
> **当前阶段：开发期** — 正式设计以 **§2.1 方案 B**（`factor_value_files` 为权威、策略按执行域查表）为准，避免生产后再拆双写。  
> 本文与 `docs/因子侧真分域因子值_设计与落地步骤.md`、`docs/多领域因子回测改造_落地步骤.md` 互补；**以代码为准**处已对照 `backtest_io_runner.py` 写明。

## 1. `backtest_io` 与 `factor_values_path`（历史行为 vs 当前代码）

实现文件：`src/backtest_io/backtest_io_runner.py`。

**当前（P0-1 已落地）**：`backtest_io` **不再**更新 `factor_files.factor_values_path`；批量路径以 **`factor_value_files`** 为准，可选跑 `sync_factor_values_path_runner.py` 回写缓存列。

**历史行为（文档留存，便于对照旧库）**：

1. **原调用时机**  
   每处理一个 `BacktestResult`，在插入 `factor_backtest` 之后，曾调用 `_upsert_factor_values_path(..., universe=res.test_universe)`（**已删除**）。

2. **查找规则**  
   - 目录：`{factor_output_dir}/by_universe/{UNIVERSE}/`（`UNIVERSE` 来自 `_safe_universe_file_tag(universe).upper()`）。  
   - 通配：`{factor_id}_{UNIVERSE}_*.csv`，多个匹配时按文件名排序取 **最后一个**（视为最新）。

3. **写库语义**  
   `INSERT ... ON CONFLICT (factor_id) DO UPDATE SET factor_values_path = EXCLUDED.factor_values_path`  
   → **`factor_files` 每因子仅一行，`factor_values_path` 会被本次回测域对应的 CSV 路径整列覆盖**，没有「仅更新 ALL」「合并多域」等分支。

4. **推论（与策略接入强相关）**  
   - 若策略执行约定只读 **全市场（ALL）** 批量 CSV，则 **最后一次跑 `backtest_io` 的 `test_universe` 若不是 ALL**，会把 `factor_values_path` 指到 HS300 等域，**策略侧会在不知情的情况下读错域**。  
   - 多域批跑时，**运行顺序决定指针**：同一因子先跑 ALL 再跑 HS300，最终指针在 HS300；反之亦然。  
   - 找不到 CSV 时仅打 warning，**不更新**该列（本次跳过 `factor_values_path` 的 UPSERT 路径——实际为函数直接 `return`，整次 for 循环里该因子若仅触发「找不到」则不会执行 UPDATE；若行不存在也不会 INSERT 带路径的空行问题需注意：当前 INSERT 带 `factor_values_path`，若 return 则该因子本轮不写 factor_files——若已有行则保持旧值）。

5. **与 `normalize_universe_code` 的差异（风险点）**  
   `_upsert_factor_values_path` 使用 `_safe_universe_file_tag`，**未**走 `common.universe_service.normalize_universe_code`（例如历史 `ALL_A` → `ALL`）。若配置或 DB 仍写 `ALL_A`，而磁盘目录为 `by_universe/ALL/`，可能出现 **找不到 CSV、指针不更新或长期错误** 的情况。收尾时应统一。

## 2. 目标状态（建议契约）

| 角色 | 约定 |
|------|------|
| **多域实证** | `factor_backtest` 多行 + `factor_universe_status`；JSON / CSV 按 `by_universe/{U}/`。 |
| **多域批量路径索引** | `factor_value_files` 按 `(factor_id, universe, …)` 维护（引擎已写 batch_csv）。 |
| **策略执行用批量快照** | **正式设计（推荐）**：策略按 `execution_universe` **查 `factor_value_files`**（`artifact_type=batch_csv`），不再依赖「回测最后一次写谁」的单行指针。 |
| **单行缓存（可选）** | `factor_files.factor_values_path` 仅作 **兼容/运维缓存**：由独立步骤从 `factor_value_files` 按默认执行域（如 ALL）**同步回填**，**不由 `backtest_io` 按本次 `test_universe` 覆盖**（避免与多域实证混淆）。 |
| **日更** | `factor_values_path_daily` 优先；空则回退批量路径（见策略 loader 实现）。日更是否分域见 P3。 |

### 2.1 开发阶段正式取向（P0-A vs P0-B）

- **推荐作为目标架构：方案 B**  
  - **权威**：多域批量 CSV 路径以 **`factor_value_files`** 为准（因子引擎已写入）。  
  - **`backtest_io` 职责**：只负责 `factor_backtest`、JSON、`factor_basic` 等；**不更新** `factor_values_path`（或删除 `_upsert_factor_values_path` 调用）。  
  - **策略工厂**：加载因子值路径时 **`SELECT … FROM factor_value_files`**，`universe = strategy.parameters.execution_universe`（归一后），并约定「最新一行 / 指定区间」规则。  
  - **可选**：单独 `sync_factor_files_from_value_files` 任务（或并入 `selection`）把执行域路径写入 `factor_values_path`，供仍读旧列的脚本过渡。

- **方案 A（仅过渡）**：在 `backtest_io` 加 `execution_universe_for_factor_files_pointer`，只有匹配时才更新 `factor_values_path`。适合赶窗口止血；**与「分得清」的长期目标不一致，不建议作为最终设计**。

## 5. P0 验收操作步骤（测试步骤）

以下在 **测试库 / 本机** 执行即可；路径以仓库根为准。

### 5.1 因子库：`backtest_io` 不再写 `factor_values_path`

1. 跑完一次 `backtest_io`（任意 `test_universe`）。  
2. 在 DB 执行（替换 `:fid` 为本次涉及的 `factor_id`）：

```sql
-- 跑 backtest_io 前后各查一次；预期：跑完后 factor_values_path 不因本次回测被改写
-- （除非你另外跑了 sync_factor_values_path_runner）
SELECT factor_id, factor_values_path
FROM factor_files
WHERE factor_id = :fid;
```

3. 确认 `factor_backtest` 仍有新行，`result_json_rel_path` 指向 `backtest_results/by_universe/...`。

### 5.2 因子库：`factor_value_files` 与磁盘

```sql
SELECT factor_id, universe, artifact_type, rel_path, created_at
FROM factor_value_files
WHERE artifact_type = 'batch_csv'
ORDER BY factor_id, universe, created_at DESC
LIMIT 20;
```

对照磁盘：`{repo}/factor_values/by_universe/{universe}/` 下是否存在对应 CSV。

### 5.3 同步脚本（可选缓存列）

```text
# 在 qclaw_factor_engine 根目录，使用已配置的 config.ini
python src/backtest_io/sync_factor_values_path_runner.py
```

再查 `factor_files.factor_values_path`：应对齐 `factor_value_files` 中 `universe = sync_factor_values_execution_universe` 的 `rel_path`。

### 5.4 策略工厂：优先读 `factor_value_files`

**前置**

1. `qclaw_strategy_engine/src/strategy_backtest/config.ini`（或你实际使用的 `config.ini`）：  
   - `[data_source]` **`mode = real`**（不是默认的 `db`）。  
   - `[factor_engine]` **`root_dir`** = **因子工程仓库根目录**（与 `factor_value_files.rel_path` 拼接后能指到真实 CSV，例如 `D:/.../qclaw_factor_engine`）。
2. `factor_db_config.ini`（`strategy_backtest` 里 `[database_factor] factor_db_config` 指向的文件）：**host/db/user** 与因子库一致，能访问 `factor_value_files`、`factor_files`。
3. 策略配置 JSON（`parameters`）里为本次测试因子写上 **`execution_universe`**，且与表里一致，例如 `"execution_universe": "ALL"`。  
4. 在因子库确认：这些 `factor_id` 在 **`factor_value_files`** 里已有 `artifact_type = 'batch_csv'` 且 `universe` 与 `execution_universe` 归一后一致（可先跑因子引擎 batch）。

**用例 A：走表成功（不应出现回退 warning）**

1. 任选一条待跑策略（`strategy_backtest_runner` 会从 `strategy_basic` 拉 `draft/optimized` 等），或本地临时造一个最小 `StrategyConfig` 调 `multi_factor_linear_template.run_backtest(config)`。  
2. 在 **`qclaw_strategy_engine` 仓库根**执行（路径按你环境改）：  
   `python src/strategy_backtest/strategy_backtest_runner.py`（若你项目入口是别的，用现有入口即可）。  
3. 打开日志 **`logs/strategy_backtest_runner.log`**（与厚模板共用同一 logger 配置时也在此文件）。  
4. **预期**：日志里**没有**子串 **`回退 factor_values_path`**；回测能算完（或至少因子面板合并成功，不因缺 CSV 报错）。

**用例 B：回退路径（应出现 warning，仍能读盘）**

1. **仅在测试库**执行，选一个已在用例 A 中成功的 `factor_id`：  

```sql
DELETE FROM factor_value_files
WHERE factor_id = '你的因子ID'
  AND universe = 'ALL'          -- 与策略 execution_universe 一致
  AND artifact_type = 'batch_csv';
```

2. 确保 **`factor_files.factor_values_path`** 仍指向该因子有效 CSV（可先跑因子库的 `sync_factor_values_path_runner`，或手工保证列非空）。  
3. 再跑同一次策略回测。  
4. **预期**：日志中出现 **`未在 factor_value_files 找到 universe=... 的 batch_csv，回退 factor_values_path`**；流程仍成功（与磁盘文件一致）。

**用例 C：只测「加载路径」不写完整回测（可选）**

在 `qclaw_strategy_engine` 下用短脚本（仓库根为 cwd、`PYTHONPATH` 含 `src`）调用：

```python
from strategy_backtest.templates.multi_factor_linear_template import _load_factor_values_paths

ids = ["JQ_ALPHA_001"]  # 换成你库里有表记录的 factor_id
p = _load_factor_values_paths(ids, path_source="eval", execution_universe="ALL")
print(p)
```

有 `factor_value_files` 时应直接得到 `rel_path`；删掉表行后应仍能 `print` 出路径但运行全链路时会打 warning（若你在外层再调 `_load_factor_panel_from_files`）。

### 5.5 域归一（`ALL_A`）

1. 若 `backtest_core` 仍配置 `test_universe = ALL_A`（不推荐），跑 `backtest_io` 后查：

```sql
SELECT factor_id, test_universe FROM factor_backtest ORDER BY id DESC LIMIT 5;
```

**预期**：入库为 `ALL`，JSON 落在 `backtest_results/by_universe/ALL/`。
