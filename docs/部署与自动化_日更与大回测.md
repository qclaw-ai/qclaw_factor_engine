## 部署与自动化：qclaw_factor_engine（日更线 + 单因子大回测）

> 目标：在云服务器上部署 `qclaw_factor_engine`，并通过定时任务实现：
>
> - **每天**跑一遍“行情 + Calendar 同步 + 因子日更”的日内 pipeline；
> - **每周**跑一遍单因子大回测（`backtest_core_runner`，可视为 S0/S1 前置评估）。

---

### 1. 环境准备

- **系统建议**：Ubuntu 20.04+/Debian 11+，已安装 Python 3.10+。
- **数据库**：可连通的 PostgreSQL（`db_factor`），表结构按照 `db_conventions.md` 中的约定建好：
  - `stock_daily`
  - `factor_basic`
  - `factor_value_files`
  - 其他与日更/回测相关的表。
- **代码部署路径建议**：
  - 根目录：`/data/qclaw/qclaw_factor_engine`

#### 1.1 拉仓库 & 安装依赖

```bash
# 1) 拉代码（示例用 SSH，自行替换）
git clone git@your-git-server:qclaw/qclaw_factor_engine.git /data/qclaw/qclaw_factor_engine

cd /data/qclaw/qclaw_factor_engine

# 2) 建议使用 venv
python3 -m venv .venv
source .venv/bin/activate

# 3) 安装依赖
pip install -r requirements.txt
```

#### 1.2 环境变量

- 统一约定：**非生产环境**使用 `config_dev.ini`，生产可用独立配置文件或设置 `ENV=prod`。

```bash
export ENV=prod
export PYTHONPATH=/data/qclaw/qclaw_factor_engine/src
# 可选：覆盖 pipeline 使用的 Python（默认 /home/ubuntu/miniconda3/bin/python，供 cron 无 PATH 时使用）
# export PYTHON_BIN=/home/ubuntu/miniconda3/bin/python
```

> 按需将上述变量写入 `/etc/profile.d/qclaw_factor_engine.sh` 或专用用户的 shell 启动脚本。

- **`PYTHON_BIN`（可选）**：`bin/run_daily_factor_pipeline.sh`、`bin/run_weekly_factor_pipeline.sh` 内默认使用 **`/home/ubuntu/miniconda3/bin/python`** 调用解释器，避免 **cron 环境找不到 `python` 命令**。Miniconda 安装路径不同时，请设置 `export PYTHON_BIN=...` 或在 crontab 同任务行前声明 `PYTHON_BIN=...`。

### 2. 配置说明（云环境推荐设置）

#### 2.1 行情 & Calendar 同步：`src/data_ingest/config_dev.ini`

关键段落：

```ini
[database]
db_host = <postgres_host>
db_port = 5432
db_user = <postgres_user>
db_password = <postgres_password>
db_name = db_factor

[jq]
user = <your_jq_username>
password = <your_jq_password>

[data_ingest]
universe = ALL_A or HS300 等
stock_codes =

[mysql_source]
enabled = false
host = <mysql_host>
port = 3306
user = <mysql_user>
password = <mysql_password>
db_name = <mysql_db>
dailyquote_table = DailyQuote
calendar_table = Calendar
dailyquote_tradedate_format = date      ; 或 yyyymmdd
calendar_tradedate_format = date        ; 或 yyyymmdd
symbol_filter = true
stock_sync_chunk_days = 3
calendar_sync_mode = all                ; all / range
```

云环境建议：

- 有外部 MySQL 源库时：
  - `mysql_source.enabled = true`，优先走 MySQL，同步失败时自动回退聚宽；
  - `symbol_filter = true` 时会按 universe 做 Symbol IN 过滤，降低同步量。
- 只用聚宽时：
  - `mysql_source.enabled = false`，只需要配置好 `[jq]` + `[data_ingest]`。

#### 2.2 日更线：`src/daily_factor_values/config_dev.ini`

关键段落（详细含义见模块 README）：

```ini
[database]
db_host = <postgres_host>
db_port = 5432
db_user = <postgres_user>
db_password = <postgres_password>
db_name = db_factor

[daily]
; 向前拉取 stock_daily 的自然日长度（rolling/REF 需要历史窗口）
lookback_days = 380
; all_in_basic（执行日更默认） | valid_only（研究/降本）
scope = all_in_basic
; 日更所属域（支持 ALL_A 自动归一为 ALL）
universe = ALL
```

云环境建议：

- **`scope`**：生产建议用 `all_in_basic`，尽量覆盖所有在 `factor_basic` 中登记且有公式的因子；
- **`lookback_days`**：380 足够支持年内大部分 rolling 因子，如有更长窗口可适当调大；
- **`universe`**：
  - 若策略工厂主要在 ALL 域执行：建议 `ALL`；
  - 若后续需要 HS300 等其他域，可在日更 cron 中为不同域分别跑一遍。

#### 2.3 单因子大回测：`src/backtest_core/config_dev.ini`

参考：

```ini
[database]
db_host = <postgres_host>
db_port = 5432
db_user = <postgres_user>
db_password = <postgres_password>
db_name = db_factor

[backtest]
; horizon（单位：交易日）
horizon = 5
n_quantiles = 10

; 因子 CSV 根目录（由 factor_engine 输出）
factor_output_dir = factor_values

; 可选：仅回测这些因子ID（逗号分隔）；留空则对目录下所有 CSV 扫描
factor_ids =
```

云环境建议：

- `factor_output_dir` 建议指向统一的 by_universe 目录根，比如：
  - `factor_values`（内部约定再使用 `by_universe/ALL` 等子目录）；
- 若因子数较多，可按需将 `factor_ids` 限制为增量因子集，或者拆分不同 cron 任务分批回测。

### 3. 手动执行命令（用于联调/排障）

#### 3.1 日内因子 pipeline（行情 + Calendar + 日更）

```bash
cd /data/qclaw/qclaw_factor_engine
source .venv/bin/activate

# 以 2026-03-30 为 T，手动跑一遍完整 pipeline
bin/run_daily_factor_pipeline.sh 2026-03-30
```

也可以不传入参数，脚本会默认使用“今天”的日期：

```bash
bin/run_daily_factor_pipeline.sh
```

如需只联调日更部分（跳过 pipeline），仍可直接调用：

```bash
python src/daily_factor_values/daily_factor_values_runner.py \
  --trade-date 2026-03-30 \
  --universe ALL \
  --scope all_in_basic
```

> 该 runner 默认读取 `src/daily_factor_values/config.ini`，并由 `common.Config` 按 `ENV` 自动切换到 `config_dev.ini`（非 prod）。

#### 3.2 单因子大回测（ALL 域）

```bash
cd /data/qclaw/qclaw_factor_engine
source .venv/bin/activate

python src/backtest_core/backtest_core_runner.py
```

如需限制回测因子：

```bash
python src/backtest_core/backtest_core_runner.py
```

> `backtest_core_runner.py` 当前不提供 `--config/--factor-ids` 参数；请通过修改 `src/backtest_core/config.ini`（非 prod 自动切 `config_dev.ini`）控制 `factor_ids/test_universe` 等配置。

### 4. 定时任务设计（cron 示例）

> 假设使用 Linux 系统级 cron，执行用户为 `qclaw`（拥有工程目录与日志写权限）。

**cron 与 Python**：`bin/` 下日更/周更脚本已通过环境变量 **`PYTHON_BIN`**（默认 Miniconda 绝对路径）调用 Python，一般无需在 crontab 里再配置 `PATH`。若需覆盖，可在对应任务前增加一行 `PYTHON_BIN=/your/path/bin/python`（与 `ENV=prod` 写法相同）。

#### 4.1 每日行情 + 日更（建议：收盘后跑）

示例：**每个交易日 20:10** 触发 ALL 域日内 pipeline，`T` 取当天日期。

1. 设置执行权限：

```bash
chmod +x /data/qclaw/qclaw_factor_engine/bin/run_daily_factor_pipeline.sh
```

2. 编辑 crontab（`crontab -e`，以 `qclaw` 用户）：

```cron
ENV=prod
10 20 * * 1-5 /data/qclaw/qclaw_factor_engine/bin/run_daily_factor_pipeline.sh
```

> 后续可以把 `1-5` 换成基于交易日历的调度；如需对 HS300 单独日更，可为 pipeline 脚本加参数或另写一份 HS300 版本。

#### 4.2 每周因子工厂大回测全链路（建议：周末跑）

示例：**每周日 03:00** 跑一次因子工厂全链路（生成因子值 batch_csv → 回测写库 → 入库筛选 → 相关性矩阵）。

1. 赋予执行权限：

```bash
chmod +x /data/qclaw/qclaw_factor_engine/bin/run_weekly_factor_pipeline.sh
```

2. crontab 中添加：

```cron
ENV=prod
0 3 * * 0 /data/qclaw/qclaw_factor_engine/bin/run_weekly_factor_pipeline.sh
```

> 若因子数很多、单次跑不完，可以在各模块 `config.ini` 中通过 `factor_ids` 做分批，或在调度层拆分多个任务。

补充说明：

- 该脚本会依次调用：
  - `factor_engine`（生成 `factor_value_files(batch_csv)` 并写出 `factor_values/by_universe/...`）
  - `backtest_io`（内部调用 `backtest_core` 计算指标，并写入 `factor_backtest` + 回测 JSON）
  - `selection_and_store`（按阈值更新 `factor_basic.is_valid` / `factor_universe_status` / `factor_files`）
  - `factor_corr`（对 `is_valid=true` 因子生成相关性矩阵并写入 Redis）
  - 上述模块均默认读取各自模块目录下的 `config.ini`，并由 `common.Config` 按 `ENV` 自动切换到 `config_dev.ini`。

#### 4.3 （可选）仅单因子回测核心（backtest_core）

如果你只想跑“回测指标计算”而不写库/不更新相关性，可直接执行：

```cron
ENV=prod
0 2 * * 0 cd /data/qclaw/qclaw_factor_engine && /home/ubuntu/miniconda3/bin/python src/backtest_core/backtest_core_runner.py
```

> crontab 里请使用 **Python 绝对路径**（按服务器上 Miniconda 实际路径修改）。更省事可直接跑 `bin/run_weekly_factor_pipeline.sh`（脚本内已用 `PYTHON_BIN` 调用各子任务）。

### 5. 监控与排障建议

- **日志位置**：
  - 日更核心日志：`logs/daily_factor_values_runner.log`
  - cron 任务追加日志：
    - `logs/cron_daily_factor_values.log`
    - `logs/cron_backtest_core.log`
- **常见问题**：
  - `factor_docs 未解析到任何因子` → 检查 `factor_docs` 目录和数据库 `factor_basic`；
  - `stock_daily 无数据` → 检查行情同步链路，确认该时间区间内有数据；
  - `factor_value_files` 无 `daily_csv` 记录 → 检查 `scope` / `universe` / `factor_ids` 设置。

--- 

### 6. 与策略工厂的衔接

- S4/S6 真实回测与执行会通过 `factor_value_files`（`artifact_type=batch_csv/daily_csv`）读取路径。
- **在部署策略工厂（`qclaw_strategy_engine`）前**，应保证：
  - 日更 cron 至少连续跑过一段时间，`factor_values/daily/by_universe/...` 下有足够日期；
  - 每周因子工厂大回测全链路至少完成一轮（生成 batch_csv + 写回测结果 + 入库筛选 + 相关性矩阵），为 `strategy_factor_list` 提供：
    - `factor_basic.is_valid=true` 的有效因子集合；
    - Redis 中最新的 `factor:corr:{UNIVERSE}:latest` 相关性矩阵。
