# 多领域因子大回测改造 — 落地步骤

> 目标：同一 `factor_id` 在 **多个股票池（领域）** 上各自保留完整大回测指标；**因子截面数值仍维持全市场一份**；策略在全市场执行时，可按条引用「该因子在哪个领域上的实证结果」作为元数据。  
> 范围：`qclaw_factor_engine`（主）与 `qclaw_strategy_engine`（配置/清单/约定对齐）。  
> 本文档为实施清单，按阶段顺序执行；每阶段完成后可做联调与验收。  
> **已定案并已落地（2026-03-24）**：`factor_universe_status` + `factor_basic.is_valid` 派生；因子清单 `factor_id` 唯一且必填 `evidence_universe`（可用 `test_universe` 别名）；迁移脚本 `sql/migrations/001_factor_multiverse_backtest.sql`。

---

## 0. 原则与术语（实施前全员对齐）

### 0.1 两条线

| 概念 | 含义 | 存储 |
|------|------|------|
| **因子值（截面/时序）** | 每个交易日、每只股票上的因子数值 | **全市场一份**（不按领域拆多份 CSV） |
| **因子大回测（实证）** | 在指定股票池上算出的 IC、夏普、回撤等 | **按领域多行**：`(factor_id, test_universe, …)` |

### 0.2 领域编码 `test_universe`

- 在配置、DB、JSON 中使用 **同一套枚举字符串**（示例，可按现有 `data_ingest` / 文档统一）：`ALL`、`STOCK`、`HS300`、`ZZ500`、`CUSTOM`…  
- 历史兼容：`ALL_A -> ALL`（读取兼容，新增写入不再使用 `ALL_A`）。  
- **禁止**同一语义多种写法混用（如「沪深300」「HS300」「000300」混存）；若需展示名，单独加 `display_name` 或映射表（可选，后期再做）。

### 0.3 与现有 MVP 的冲突点（已改）

- 已在 `factor_backtest` 增加 `test_universe`、`result_json_rel_path`；`selection_and_store` 改为 `DISTINCT ON (factor_id, test_universe)`。  
- `factor_universe_status` 承载按领域 `is_valid`；`factor_basic.is_valid` = 是否存在任一领域为 TRUE（供日更等兼容读取）。  
- `db_conventions.md` 中「实验/多版本用新 factor_id」仍适用 **公式版本**；**多领域**不靠拆 `factor_id`。

## 1. 阶段一：数据库（因子库）

**库**：`qclaw_factor_engine/sql/` 新增迁移脚本（勿只改 `schema_mvp.sql` 而不迁移现网；建议 `migrations/00x_factor_backtest_universe.sql`）。

### 1.1 `factor_backtest` 表

1. 增加列：`test_universe VARCHAR(64) NOT NULL`。  
2. **历史数据回填**（见 §7）：若现有行无法获知领域，用约定默认值（如 `LEGACY` 或 `ALL`），并在注释中说明；新数据必须从回测任务写入真实领域。  
3. 索引与唯一性（二选一，按你是否允许「同因子、同领域、同区间、同 horizon」重复跑）：  
   - **推荐**：普通索引 `(factor_id, test_universe, backtest_time DESC)`，用于「取某因子在某领域最新一条」；  
   - 若需强约束重复提交：再加 **部分唯一** 或业务唯一 `(factor_id, test_universe, backtest_period, horizon)`（需确认 `backtest_period` 字符串是否稳定）。  
4. （可选）`result_json_rel_path VARCHAR(1024)`：该次领域回测产物 JSON 的相对路径，避免 `factor_files` 单行无法挂多域报告。

### 1.2 `factor_basic` 表（已定案）

1. `test_universe` 列保留为 **md/默认提示**；权威实证以 `factor_backtest` + `factor_universe_status` 为准。  
2. 已新增 **`factor_universe_status(factor_id, test_universe, is_valid, updated_at)`**，主键 `(factor_id, test_universe)`。  
3. **`factor_basic.is_valid`**：`selection_and_store` 在写入各域状态后，按 **「任一领域 `is_valid=TRUE`」** 同步到 `factor_basic`（兼容 `daily_factor_values` 等只查 `factor_basic.is_valid` 的逻辑）。

### 1.3 `factor_files` 表

- **因子 md、全市场日更路径**：仍可 **一行 per `factor_id`**（与「全市场一份因子值」一致）。  
- 若各领域大回测 JSON 都要留盘：优先挂在 **`factor_backtest.result_json_rel_path`**；若坚持集中表，再考虑 `factor_files` 拆成 `(factor_id, file_scope)` 或子表（复杂度上升，非必须）。

### 1.4 外键与脚本

- 确认 `factor_backtest.factor_id → factor_basic.factor_id` 仍成立。  
- 在 `docs/db_conventions.md` 增加一节：**多领域回测与 `factor_basic` 字段语义**。

## 2. 阶段二：回测写入链路（`qclaw_factor_engine`）

### 2.1 `backtest_core` / 回测入口

1. 回测任务配置（ini / CLI / 调用参数）增加 **`test_universe`**（必填，新任务）。  
2. 回测在计算 IC/分层等指标时：在内存中对 **全市场因子值 + 收益** 按当日成分（或配置规则）做 **mask**，领域由 `test_universe` 解析（与 `data_ingest` 成分解析逻辑复用或抽公共模块）。  
3. 输出结构体 `BacktestResult`（或等价）携带 `test_universe`。

### 2.2 `backtest_io_runner`（或等价写库模块）

1. `INSERT factor_backtest` 增加 `test_universe`（及可选 `result_json_rel_path`）。  
2. `_ensure_factor_basic`：  
   - `ON CONFLICT DO NOTHING` 保留；**不要**用单次 md 里的 `test_universe` 覆盖已有多领域策略下的语义（若需更新 `default_evidence_universe`，单独 `UPDATE` 规则写清）。  
3. 批跑：同一 `factor_id` 对 `test_universe ∈ {HS300, ZZ500, ALL}` 循环调用，产生多行。

### 2.3 批处理与运维

1. 文档化：**领域批跑顺序**、失败重试是否产生重复行、如何用 `(factor_id, test_universe, backtest_time)` 取最新。  
2. 日志中打印 `factor_id + test_universe`，便于对账。

---

## 3. 阶段三：筛选与入库（`selection_and_store_runner`）

1. 加载「最新回测」的 SQL 从 `DISTINCT ON (factor_id)` 改为 **`DISTINCT ON (factor_id, test_universe)`**（`ORDER BY factor_id, test_universe, backtest_time DESC, id DESC`）。  
2. 阈值判断：**按领域**计算 `pass_standard`；更新 `factor_backtest.pass_standard` 时 **精确定位到对应 `id` 或 (factor_id, test_universe, 最新 id)**。  
3. 更新 `factor_basic.is_valid`（或 `factor_universe_status`）时遵守 §1.2 选定规则；若按领域独立有效，**禁止**用「领域 A 的结果」更新「全局 valid」而不加说明。  
4. 配置：`factor_threshold_config.scene` 是否需要按领域区分（如 `A_stock_daily_single_factor_HS300`）；若共用一个 scene，文档说明 **同一阈值应用于所有领域** 的产品含义。

---

## 4. 阶段四：日更与其它读库模块

### 4.1 `daily_factor_values_runner`

- **不按领域拆输出**（坚持全市场一份）：逻辑可不变，仅确认读取 `stock_daily` 覆盖全市场可交易集。  
- 配置与文档写明：**日更 universe = 全市场**，与 **大回测 universe** 分离。

### 4.2 `factor_corr_matrix` 等

- 相关性矩阵是否在 **某领域子集** 上算：增加参数 **`universe`**；默认建议 `ALL` 或与策略执行域一致。  
- 从 DB 取因子列表时：若仍 `WHERE is_valid`，需明确该 `is_valid` 对应 §1.2 哪条规则。

### 4.3 `factor_docs` / Markdown

- 头部「适用股票池」可保留为 **文档说明**；**不以 md 为唯一真相**覆盖 DB 里多行 `factor_backtest`。可选：解析为 `suggested_universe`，仅作默认回测领域提示。

---

## 5. 阶段五：策略工厂（`qclaw_strategy_engine`）

### 5.1 因子清单 JSON（`factor_lists/*.json`）

1. 每条因子记录增加或明确：  
   - `evidence_universe`（或沿用 `test_universe` 字段但 **语义定为「实证领域」**）：表示 **IC 等指标来自哪条 `factor_backtest`**。  
2. 生成清单的流水线（从 DB 导出）：对每个 `factor_id` 可按需 **选一条** 领域记录（如业务默认 `ALL`，或「过阈的最高夏普领域」），并写入清单；**禁止**无声混用不同领域却不落字段。

### 5.2 策略配置 draft / `strategy_config`

1. 增加 **`execution_universe`**（或 `trading_universe`）：固定为 `ALL` / 全市场 时显式写出，便于回测与审计。  
2. LLM / 人工构建策略时：prompt 与 schema 要求输出 **每条因子的 `evidence_universe`**（与权重并列）。

### 5.3 `strategy_backtest` 与 runner

1. 策略回测写库时：`test_universe` 列表示 **策略执行/净值核算股票池**（应与 `execution_universe` 一致）。  
2. 策略侧 **只读** 因子库时：若需展示「因子来自哪领域实证」，读 `factor_backtest` 或因子清单字段，**不要**用策略的 `test_universe` 冒充因子实证域。

### 5.4 文档

- 更新 `qclaw_strategy_engine/docs/db_conventions_strategy.md` 与《策略工厂自动化流程》相关章节：**执行域 vs 因子实证域**。

---

## 6. 阶段六：联调与数据回填（`§7` 详述）

### 6.1 最小联调用例

1. 选一测试因子 `F_TEST`，对 `HS300`、`ZZ500` 各跑一次大回测入库。  
2. 查库：`SELECT factor_id, test_universe, ic_value, pass_standard FROM factor_backtest WHERE factor_id='F_TEST' ORDER BY test_universe, backtest_time DESC` 至少两行。  
3. 跑 `selection_and_store`：两领域 `pass_standard` 独立正确。  
4. 导出因子清单：`factor_id` **全局唯一**，每条必填 `evidence_universe`（或别名 `test_universe`）；`load_factor_list` 会校验。  
5. 全市场策略回测：使用全市场因子值 CSV，净值与清单领域标注一致。

### 6.2 清单内同一 `factor_id` 是否允许多条（已定案）

- **不允许**；已在 `builder_common.load_factor_list` / `strategy_config_builder_human._load_factor_list` 去重并校验 `evidence_universe`。

---

## 7. 历史数据迁移

1. `factor_backtest` 增加 `test_universe` 后，对旧行执行 `UPDATE ... SET test_universe = '<约定>' WHERE test_universe IS NULL`（若用 `NOT NULL`，需在默认值后再改为业务值）。  
2. 若旧数据实际只在 HS300 上跑过，将约定设为 `HS300` 比 `ALL` 更诚实（需业务确认）。  
3. 迁移后跑一次 `selection_and_store` 在**测试库**验证，再生产。  
4. 备份：迁移前 `pg_dump` 或等价。

## 9. 风险与回滚

- **风险**：旧代码未传 `test_universe` 导致写入失败或领域错误 → 用集成测试与 staging 先跑。  
- **回滚**：保留迁移前备份；新列可降级为可空并停写，恢复旧 `selection` SQL（不推荐长期双轨）。
